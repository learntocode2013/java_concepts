AtomicReference in Lock-Free Programming

A practical tour: the CAS mental model, the canonical patterns, the pitfalls, and where it sits relative to synchronized and ReentrantLock.

AtomicReference<V> — what it is

A wrapper around a single object reference that exposes hardware-level atomic operations on it: get, set, and crucially compareAndSet(expected, new). The CAS operation compiles down to a CPU instruction (CMPXCHG on x86, LDXR/STXR on ARM) — no JVM monitor, no OS park, no context switch. That's the foundation of lock-free programming on the JVM.

The mental model:

boolean compareAndSet(V expected, V update) {
    if (currentValue == expected) {   // atomic, in CPU silicon
        currentValue = update;
        return true;
    }
    return false;
}

The check-and-write happens as a single uninterruptible step. No thread can observe a half-updated state, and no thread can sneak in between the read and the write.

The “rebase” loop — the standard lock-free idiom

You almost never call compareAndSet once. You call it in a retry loop:

AtomicReference<Snapshot> ref = ...;
while (true) {
    Snapshot current = ref.get();       // 1. observe
    Snapshot updated = derive(current); // 2. compute new state, pure function
    if (ref.compareAndSet(current, updated)) {
        break;                           // 3. publish — success
    }
    // 4. someone else won the race; loop and try again with their new state
}

The pattern: read → derive → CAS → retry if stale. This is the “rebase” model — like a git rebase where you re-do your work against whatever the latest state is. The thread never blocks; it just keeps recomputing. Java 8+ wraps the loop for you in updateAndGet / accumulateAndGet:

ref.updateAndGet(current -> derive(current));

Example 1: lock-free lazy initialization

Naive version with a lock would synchronize the whole method. Lock-free:

public final class Cache<T> {
    private final AtomicReference<T> value = new AtomicReference<>();
    private final Supplier<T> factory;

    public Cache(Supplier<T> factory) { this.factory = factory; }

    public T get() {
        T existing = value.get();
        if (existing != null) return existing;          // fast path: already cached
        T candidate = factory.get();                    // possibly expensive
        return value.compareAndSet(null, candidate)
                ? candidate                             // we won — publish ours
                : value.get();                          // someone else won — use theirs
    }
}

What's lock-free about this: zero blocking. Two threads can race; one wins the CAS, one wastes the factory.get() call. No one waits. The contention cost is “occasionally compute twice” instead of “always wait.”

When factory.get() is expensive and concurrent calls are rare, this is the wrong trade. When the value is small / cheap to compute and threads pile in, it crushes a synchronized if (value == null) ... block.

Example 2: lock-free Treiber stack (the canonical example)

This is Exercise 2 from your Devoxx notes. A stack where push and pop never block:

public final class LockFreeStack<E> {
    private static final class Node<E> {
        final E value;
        final Node<E> next;
        Node(E value, Node<E> next) { this.value = value; this.next = next; }
    }

    private final AtomicReference<Node<E>> head = new AtomicReference<>();

    public void push(E value) {
        Node<E> oldHead;
        Node<E> newHead;
        do {
            oldHead = head.get();
            newHead = new Node<>(value, oldHead);
        } while (!head.compareAndSet(oldHead, newHead));
    }

    public E pop() {
        Node<E> oldHead;
        Node<E> newHead;
        do {
            oldHead = head.get();
            if (oldHead == null) return null;
            newHead = oldHead.next;
        } while (!head.compareAndSet(oldHead, newHead));
        return oldHead.value;
    }
}

What makes this safe:

  • The Node is immutable (both fields are final). Once published, no one mutates it, so readers seeing oldHead.next always see a consistent next pointer.
  • The CAS on head is the single linearization point. The reader of head either sees the value before your push or after — never half-pushed.
  • The new node is constructed outside the CAS, so even if you retry, the cost is just allocation, not a held lock.

What threads do under contention: they spin. There's no parking, no kernel transition, no ObjectMonitor. That's the dramatic perf advantage you saw between synchronized (3.2 s) and ReentrantLock (0.4 s); a CAS-based version would beat both at low-to-moderate contention.

Example 3: a versioned snapshot (state machine update)

When you want to atomically replace a compound state — e.g. a configuration bundle:

public final class Config {
    final String host;
    final int port;
    final Duration timeout;
    Config(String h, int p, Duration t) { host = h; port = p; timeout = t; }
}

private final AtomicReference<Config> current = new AtomicReference<>(initial);

public void updatePort(int newPort) {
    current.updateAndGet(c -> new Config(c.host, newPort, c.timeout));
}

public Config snapshot() {
    return current.get();   // readers never block, never see a torn state
}

A reader getting current.get() always sees a fully-constructed Config. Writers race but only one wins each round; losers re-derive from the latest. No lock, no volatile discipline beyond what AtomicReference already gives you.

This is how libraries like Caffeine, Disruptor, and many event-loop engines manage their hot state.

Example 4: lock-free idempotent flag

Sometimes you just want “exactly one thread enters this block, the rest skip”:

private final AtomicReference<State> state = new AtomicReference<>(State.NEW);

enum State { NEW, STARTED, STOPPED }

public void start() {
    if (state.compareAndSet(State.NEW, State.STARTED)) {
        actuallyStart();  // exactly one caller reaches here
    }
}

A single CAS is enough — no loop needed because the transition is one-shot. This pattern shows up everywhere: shutdown latches, one-time initialization, leader election within a process.

The ABA pitfall

CAS asks “is the value still expected?” but it can't tell the difference between “unchanged” and “changed away and then changed back.” Consider:

  1. Thread A reads head → node X.
  2. Thread A computes newHead = X.next.
  3. Thread B pops X, pops its successor, then pushes a new node that happens to reuse X's memory — now head is X again, but the rest of the list is different.
  4. Thread A's CAS succeeds — head == X — but newHead = X.next is stale data from step 2, pointing into a list that no longer exists.

Java mitigates this somewhat (GC keeps X alive as long as A holds a reference, so memory isn't reused), but logically equivalent ABA still bites with object pooling, recycled buffers, or sequence counters.

Solution: stamp the reference with a version counter that increments on every change, so “A → B → A” is detectable as “stamp went 0 → 1 → 2”:

AtomicStampedReference<Node<E>> head = new AtomicStampedReference<>(null, 0);

int[] stampHolder = new int[1];
Node<E> oldHead = head.get(stampHolder);
int oldStamp = stampHolder[0];
// ...
head.compareAndSet(oldHead, newHead, oldStamp, oldStamp + 1);

Now the CAS fails if the stamp moved, even when the reference looks identical. (This is Exercise 3 in your Devoxx notes.)

AtomicMarkableReference is the cheaper cousin: one boolean instead of an int counter, useful when “is this node logically deleted?” is the only extra bit you need.

When NOT to reach for AtomicReference

Lock-free is not free. Consider sticking with a lock when:

  1. The critical section does more than one field's worth of work. CAS works on a single reference. If you need to atomically update three fields, you either bundle them into one immutable object (extra allocation per update) or you need transactional plumbing. A synchronized block is often cheaper.
  2. Contention is high and sustained. Under heavy contention CAS loops spin — burning CPU, generating cache-line ping-pong between cores. With dozens of writers, a fair-ish lock with parking can outperform a CAS loop. Benchmark.
  3. You need ordering across multiple variables. AtomicReference only orders accesses to itself. If you need to publish “field X then field Y”, you need a single reference to a holder object containing both, or you need explicit VarHandle fences.
  4. You'd write a single-counter increment. Use AtomicLong / LongAdder instead — they're specialized and faster (especially LongAdder under contention, which striped-counters to avoid the cache-line war).

Where it fits in your concurrency hierarchy

From your Locks_Monitors_And_Atomics.md table:

LayerMechanismWhere contention is paid
JVM monitorssynchronized / ObjectMonitorPark + stranding cost in C++
AQSReentrantLock, Semaphore, …Park via LockSupport, FIFO queue in Java
HardwareAtomic*, VarHandleSpin in CPU; cache-line bouncing

AtomicReference is the bottom row. It's the building block you'd reach for to implement the non-blocking version of anything in the upper rows — a lock-free queue (à la Michael-Scott), a lock-free set, a counter, a flag. The trade-off is always: avoid blocking by retrying optimistically, at the cost of wasted work under contention and the responsibility of getting the memory model right.
